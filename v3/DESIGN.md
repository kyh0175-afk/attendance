## 0. 요약 (한 문단)

v3는 출석의 정의를 "입실 1회"에서 **"입실+퇴실 2포인트"**로 바꾼다. 학생은 학번+PIN으로 로그인하는 **PWA**(홈 화면 설치형 웹앱)를 쓰고, **입실·퇴실 모두 교사가 발급하는 랜덤 코드**를 입력한다(입실 코드=세션 시작 시 자동 발급·세션 내내 게시, 퇴실 코드=마감 전 발급·유효 10분). **QR은 v3에서 제거** — 코드가 세션 식별과 현장 증명을 겸하고, 교사는 칠판 판서만으로 운영 가능하다. 문 앞 **키오스크(학생증 바코드 스캔, 옵셔널)**가 생기면 코드 없이 스캔만으로 입/퇴실이 된다. 입실은 있는데 퇴실이 없는 기록은 `퇴실미확인`으로 자동 마킹되어 관리자 통계에 잡힌다. 도입 첫 2주는 **측정 모드**(제재 없음, 통계만)로 운영해 실태를 확인한 후 룰을 확정한다.

## 1. 배경과 문제

- 현행(v2): 교사 QR 세션 → 학생이 폰으로 스캔 → 학번 입력 → 출석. **입실 1회 확인뿐**이라 스캔 직후 이탈해도 출석 처리.
- 운영 사실: 입실 후 **폰 제출**(통짜 박스, 슬롯 없음) → 자습 → 종료 시 폰 수령. 자유좌석. 심야 최대 58명.
- 상용 검증(딥리서치): 관리형 독서실 업계 표준 = **선언형 체크**(학생이 직접 찍는 입/퇴실/외출 + 미선언 부재를 예외로 플래그). 센서 자동감지는 상용에도 없음 → v3 방식이 검증된 패턴.

### 기각된 대안 (재제안 방지)
| 대안 | 기각 사유 |
|---|---|
| GPS/위치 앱 | 위치정보법(무동의 수집 = 5년/5천만원, 학생 **본인** 동의 필수 — 학부모 동의 갈음 불가, 2024 서울행법). 비행기모드로 회피 가능. 폰 두고 이탈은 원리상 못 잡음. 네이티브 앱 필수 |
| 중간 랜덤 재인증 | 자습 중 전원 폰 꺼내기 = 프로그램 취지 훼손. 폰 제출 정책과 충돌. 사자님 기각 |
| 좌석맵 점검 | 자유좌석이라 불성립 |
| 폰 슬롯 대조 | 보관함이 통짜 박스(번호 슬롯 아님) |
| 지문 인증 | 생체정보 = 민감정보, 별도 동의+암호화 의무. 학교 부담 과대 |
| 카드 신규 제작 | 제작 여건 없음. 대신 **기존 학생증 바코드** 활용 |
| 네이티브 앱 | 단일 학교 전용 앱은 App Store 공개 출시 거부(가이드라인 3.2). GPS 기각으로 필요성도 소멸 → PWA |

### 설계 원칙 (인권위 2025-08 결정 반영)
이탈 관리는 **기록·집계·알림까지만**. 물리적 봉쇄나 "이탈 불가" 강제로 설계하지 않는다. 정당한 이탈 경로(조퇴 처리)를 항상 제공한다. 자습 참여의 자발성 전제를 유지한다.

## 2. 사용자 흐름

### 2-1. 학생 (PWA — 폰)
```
[최초 1회] 접속 → 학번+초기PIN 로그인 → PIN 변경 → "홈 화면에 추가" 안내 → 설치
[입실]    자습실 도착 → 앱 '입실' 버튼 → 칠판의 입실 코드 4자리 입력 → 입실 완료 → 폰 제출
          (코드가 세션 식별을 겸함 — 세션 선택 UI 불필요, 서버가 활성 세션 매칭)
[자습]    (폰 없음 — 시스템 관여 없음)
[퇴실]    종료 → 폰 수령 → 앱 '퇴실' 버튼 → 교사가 안내한 퇴실 코드 4자리 입력 → 퇴실 완료
[조회]    홈 = 이번 달 출석 횟수/시간, 주간·월간 캘린더(v2 check 화면 계승), 수정·요일변경 요청
```
- 로그인 상태는 기기에 유지 → 재로그인 불필요. **본인 계정으로만 출석 가능** = 대리 출석하려면 폰을 통째로 맡겨야 하는데 폰 제출 정책과 충돌 → 자연 억제.
- 폰 미소지 학생: 키오스크(있으면) 또는 교사 화면 수동 처리(v2 수동출석 패턴 계승).

### 2-2. 교사 (태블릿/노트북 — 세션 단말)
```
[시작]  장소 선택 → 프로그램/교사명 → 세션 시작 → **입실 코드 4자리 자동 발급·대형 표시** → 칠판 판서
[진행]  실시간 명단: 입실 N명 (v2 동일). 입실 코드는 세션 활성 동안 계속 유효(지각 입실 커버)
[마감전] "퇴실 코드 발급" 버튼 (마감 10분 전 리마인드 배너) → 별도 4자리 코드 대형 표시 → 칠판/구두 안내
[마감]  미퇴실 n명 표시 → 개별 확인: [조퇴 처리(사유)] / [수동 퇴실] / [그대로 마감]
       → 마감 확정 시 잔여 미퇴실 행 = '퇴실미확인' 자동 마킹
```
- 입실 코드: 세션 시작 시 자동 생성, 세션 활성 동안 유효. **동시 활성 세션 간 중복 금지**(발급 시 서버가 보장) → 코드만으로 세션 특정. QR 완전 대체(qrcodejs 의존성 제거). 원격 공유 리스크는 v2 QR(URL 전달)과 동일 수준이며, 퇴실 체크 도입으로 "입실만 원격" 이득 자체가 소멸.
- 퇴실 코드: 세션별 랜덤 4자리(입실 코드와 다름), **발급 후 10분 유효**(1회 재발급 가능). 서버가 검증(세션·코드·유효시간·본인 입실기록 존재).
- 코드 원격공유(카톡 찍튀 2.0) 리스크: 짧은 유효창 + 본인 계정 필수 + "밖의 친구"는 자기 계정으로만 퇴실 가능(입실기록 없으면 거부) → 실질 억제. 잔여 리스크(중간이탈 후 마감 때 복귀해 직접 입력)는 **수용**(사자님 결정 7/21) — 재진입 비용 자체가 억제력이며, 반복 패턴은 통계로 드러남.

### 2-3. 키오스크 (옵셔널 모듈 — 문 앞 Windows 터치PC + 바코드 리더)
```
[입실] 학생증 바코드 스캔 → 삑 → 입실 완료 (화면에 이름 2초 표시)
[퇴실] 동일 스캔 → 퇴실 완료
```
- **코드 불필요**: 단말이 문 앞에 있으므로 스캔 = 현장 증명.
- 바코드 리더 = USB HID(키보드 에뮬) 가정 → 브라우저 입력창 포커스만으로 동작, 연동 코드 0.
- 처리량: 스캔 ~1.5초/명 → 60명 퇴실 ~2분 (넘버패드 대비 4배).
- 전제 확인(보류 중): ①바코드 내용=학번 여부 ②Edge 키오스크 모드 셋업 ③supabase.co 외부망 접근. **v3.0은 키오스크 없이도 완결** — 코드 방식이 기본, 키오스크는 가속 모듈.

### 2-4. 관리자 (v2 관리자 화면 계승 + 확장)
- 기존 8탭 유지. **분석 탭 확장**: 퇴실율, `퇴실미확인` 통계(학생별·기간별), 이상 패턴(반복 미퇴실자) 목록.
- 학생 PIN 초기화 버튼(명단 관리).

## 3. 인증 설계 — **Supabase Auth 확정 (7/21 사자님)**

원칙: **학생은 최대한 단순하게, 복잡한 작업은 운영자·시스템이** (사자님 지시).

### 학생 경험 (전부)
| 시점 | 학생이 하는 일 |
|---|---|
| 최초 1회 | 학번 5자리 + 초기 PIN 입력 → 새 PIN 설정 (~1분) |
| 이후 | 없음 — 앱 열면 자동 로그인 |
- 계정 생성·이메일·전화번호·인증문자 **없음**. 계정은 명단 업로드 시 시스템이 미리 생성.

### 운영자(사자님) 작업
1. 명단 엑셀 업로드(기존과 동일) → **"계정 일괄 생성" 버튼 1회**
2. 초기 PIN 정책 결정 1회 (권고: 생년월일 4자리 + **첫 로그인 시 변경 강제** — 학생끼리 생일을 알 수 있으므로 변경 강제는 필수)
3. 전출입: 기존 명단 비활성화 버튼 그대로 (계정 비활성 연동)
4. PIN 분실: 관리자 화면 "PIN 초기화" 버튼

### 내부 구현 (학생·운영자 비노출)
- 학생 계정 = `{학번}@st.yubongsystem.com` 가상 이메일 + PIN을 비밀번호로. 이메일 확인 비활성, 자가 가입 차단(관리자 생성만).
- 일괄 생성: Supabase Edge Function(관리자 토큰 검증) 경유 — service_role 키는 서버측에만 존재, 클라이언트 비노출.
- **채택 근거**: RLS를 `auth.uid()` 기반으로 잠금 — **v2에서 못 닫은 PII 읽기(명단·출석 전체 노출)가 이번에 닫힘**. 세션/토큰 갱신 supabase-js 자동, Realtime도 authenticated.
- (기각) 자체 토큰: 학생 경험은 동일하나 전 데이터 접근을 수제 RPC로 감싸야 해 표면 급증 + 잠금 수작업. (기각) 소셜 로그인: 학생 개인계정 강제.

교사용은 **공용 교사 PIN**(개인계정 부담 회피) → teacher 역할 로그인으로 명단 접근. 관리자는 v2 토큰 체계 유지(또는 Auth admin 역할로 통합 — W3에서 결정).

## 4. 데이터 설계

### 4-1. 기존 테이블 (유지 + 확장 — DB 연속성 원칙)
```
students        + pin 관련은 Auth로 이관(자체 저장 없음), 바코드ID 컬럼(학생증 바코드≠학번일 경우만)
attendance      + 퇴실시각 timestamptz null
                + 퇴실방식 text null  ('코드'|'키오스크'|'교사수동'|'조퇴')
                + 상태 확장: '출석' | '퇴실미확인' | '조퇴'  (기존 '출석' 유지 — 하위호환)
sessions        + 입실코드 text null           (세션 시작 시 발급, 활성 중 유효)
                + 퇴실코드 text null, 퇴실코드만료 timestamptz null
                ※ 활성 세션 간 입실코드 중복 금지 (부분 유니크 인덱스)
```
- v2 데이터는 무변경 그대로 유효 (퇴실 컬럼 null = "v2 시절 기록"으로 해석).
- '중도이탈' 대신 **'퇴실미확인'** 명칭 채택: 단정 배제(코드 안내를 못 들은 경우 등 오탐 여지) + 인권위 원칙 부합. ★결정 B

### 4-2. 새 RPC (v2 보안 패턴 계승 — 전부 SECURITY DEFINER + 권한 최소화)
```
issue_exit_code(session_id)            교사 — 코드 생성·만료 설정
check_out(session_id, code)            학생(authenticated) — 검증 후 본인 행에 퇴실 기록
kiosk_check(barcode)                   키오스크 — 입실/퇴실 토글 (키오스크 전용 role/키)
finalize_session(session_id)           교사 — 마감 + 미퇴실 행 '퇴실미확인' 마킹
mark_early_leave(session_id, 학번, 사유) 교사 — 조퇴 처리
admin_reset_pin(token, 학번)            관리자 — PIN 초기화
```
- RLS: students/attendance SELECT = 본인 행(auth.uid 매핑) + teacher/admin 역할만 전체. anon SELECT **폐지** → v2 잔여과제 종결.

**빌드 없는 멀티파일** (권고): v2와 같은 바닐라 JS이되 14.5K줄 단일파일 반복을 피한다.
```
v3/
  index.html      학생 PWA (로그인·입실·퇴실·내 현황)
```
